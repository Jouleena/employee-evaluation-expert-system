Employee Performance Expert System
Project Documentation
Date: May 15, 2026

1. Executive Summary
This repository implements an interpretable employee performance evaluation system based on fuzzy logic. The system consumes four employee metrics, applies a rule-based fuzzy inference engine, and produces a numeric score and human-readable label together with a structured explanation of the reasoning. The implementation supports three entry points: a CLI, a FastAPI backend, and a responsive browser UI.

2. Key Features
- Transparent, auditable reasoning: returns membership values, active rules, and output strengths.
- Reusable core: single fuzzy engine shared across CLI, API, and frontend.
- Multiple interfaces: terminal app (`main.py`), REST API (`api.py`), and browser UI (`index.html`).
- Test harness: predefined cases in `test_cases.py` for regression checks.

3. Architecture Overview
The repository is structured for separation of concerns:
- `fuzzy_system.py`: core fuzzy sets, rules, inference, and defuzzification.
- `api.py`: FastAPI application exposing evaluation endpoints.
- `index.html`: browser-based frontend for data entry and explanation display.
- `main.py`: command-line interface for interactive evaluation.
- `test_cases.py`: sample inputs and a test runner to validate behavior.

Data flow (high level): user input → fuzzy membership computation → rule evaluation → aggregation and defuzzification → result (score, label, explanation).

4. Fuzzy Engine (fuzzy_system.py)
4.1 Inputs
The evaluator accepts four inputs with these nominal ranges:
- `attendance`: 0–100
- `productivity`: 0–100
- `cooperation`: 0–10
- `suggestions`: 0–10

4.2 Membership functions
Each input is modeled with overlapping linguistic terms (e.g., low/medium/high) using triangular and trapezoidal membership shapes to enable partial membership across terms.

4.3 Core functions
- `get_membership(value, mf)`: computes membership degree (0–1) for a crisp input and a membership function.
- `_strongest_membership(...)`: returns the label with the highest membership degree for explanation summaries.
- `evaluate_employee(...)`: runs the full inference pipeline and returns `(score, label)` or `(score, label, explanation)` when `explain=True`.

4.4 Rule base and outputs
The rule base encodes domain heuristics (e.g., high productivity + high attendance → Excellent). Rule activations are combined into output strengths for classes like Excellent, Good, Acceptable, and Weak, then defuzzified with the centroid method to yield a numeric score.

5. API (api.py)
5.1 Endpoints
- `POST /evaluate`: accepts employee metrics, calls `evaluate_employee(..., explain=True)`, and returns `{score, label, explanation}`.
- `GET /`: serves the frontend page.

5.2 Implementation notes
The API uses FastAPI with CORS enabled for the browser UI. Input validation is performed by a Pydantic `EmployeeInput` model.

6. Frontend (index.html)
The frontend provides a responsive dashboard for entering multiple employee records, running evaluations, and viewing explanations. Key UI components include summary statistics, per-row evaluation controls, and a detailed explanation panel that renders membership values and active rules for transparency.

7. CLI (main.py)
The terminal interface mirrors the API frontend: it prompts for metrics, displays the numeric score and label, and can print the full explanation (strongest memberships, rule activations, and output strengths) when requested.

8. Testing & Validation (test_cases.py)
Predefined test cases exercise representative profiles; `run_tests()` prints results and a pass/fail status against expected labels. Note: some legacy test expectations (e.g., TC06–TC08) may not match the current rule set and should be reviewed when updating rules.

9. Dependencies & Setup
Install runtime dependencies from `requirements.txt`. Typical packages:
- `fastapi`, `uvicorn`, `numpy`, `scikit-fuzzy`, `pydantic`.

Quick start (example):
1. Create and activate a Python environment.
2. Install dependencies: `pip install -r requirements.txt`.
3. Run API server: `uvicorn api:app --reload`.
4. Open `index.html` in a browser or use the CLI: `python main.py`.

10. Design Rationale
- Explainability prioritized: the system surfaces membership degrees and rule activations so stakeholders can audit decisions.
- Single source of truth: all interfaces invoke the same fuzzy engine to ensure consistent behavior.

11. Known Issues & Next Steps
- Review and update legacy test expectations (TC06–TC08) to align with the active rule set.
- Add automated unit tests for rule coverage and edge cases.
- Consider packaging the frontend to be served statically by the API for an integrated deployment.
